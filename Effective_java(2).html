Effective Java_(2)

ITEM 2. 생성자에 매개변수가 많다면 빌더를 고려하라.

정적 팩터리와 생성자의 단점으로 선택적 매개변수가 많을 때 적절히 대응하기가 어렵다는 점이다.

예를 들면 식품 포장 영양정보를 표현하는 클래스를 생각해보자. 영양정보로 1회 내용량, 총 n회 제공량, 1회 제공량당 칼로리 등 총 20개가 넘는 선택 항목으로 이뤄진다. 그런데 대부분 제품은 이 선택 항목 중 대다수의 값이 그냥 0이다.

→ 클래스용 생성자 or 정적 팩토리 메서드에서는 점층적 생성자 패턴을 이용하여 만들곤 하였다.

1. 3점층적 생성자 패턴

public class NutritionFacts {
	private final int servingSize;
	private final int servings;
	private final int calories;
	private final int fat;
	private final int sodium;
	private final int carbohydrate;

	public NutritionFacts(int servingSize, int serving){
		return this(servingSize, servings, 0);
	}
	public NutritionFacts(int servingSize, int serving, int calories){
		return this(servingSize, servings, calories, 0);
	}
	public NutritionFacts(int servingSize, int serving, int calories, int fat){
		return this(servingSize, servings, calories, fat, 0);
	}
	public NutritionFacts(int servingSize, int serving, int calories, int fat,int sodium){
		return this(servingSize, servings, calories, fat, sodium,0);
	}

	public NutritionFacts(int servingSize, int serving,int calories, int fat, int sodium, int carbohydrate)){
		this.servingSize = servingSize;
		this.servings = servings;
		...
	}
}

하지만 매개변수가 많아지면 많아질 수록 점층적 패턴을 쓸 수도 있겠지만, 매개변수 개수가 많아지면 클라이언트 코드를 작성하거나 읽기 어렵다.

그 대안으로 자바 빈즈 패턴으로 set으로 아래와 같은 설정이 가능하다

2. 자바빈즈 패턴

NutritionFacts pepsi = new NutritionFact();
pepsi.setServingSize(40);
pepsi.setServing(8);
pepsi.setCalories(100);
pepsi.setSodium(35);
pepsi.setCarbohydrate(27);

문제점

  • 자바빈즈 패턴에서 객체 하나를 만들기 위해 메서드를 여러번 호출해야하고, 객체가 완성되기 전까지는 일관성(consistency)이 무너진 상태에 놓이게 된다.
  • 일관성이 무너지기 때문에 클래스를 불변으로 만들 수 없고 이것은 스레드 안정성을 얻기 위해서 추가 작업이 필연적이다.

두가지의 장점만 취한 방법

다행히 위의 장점만을 취한 빌더 패턴이라는 방법이다. 클라이언트는 필요한 객체를 직접 만드는 대신 필수 매개 변수 만으로 생성자를 호출하여 빌더 객체를 얻는다. 그런 다음 빌더 객체가 제공하는 일종의 setter 메서드들을 이용하여 원하는 선택 매개변수들을 설정한다. 그리고 마지막으로 build라는 메서드를 호출하여 우리가 원하는 (보통 불변의) 객체를 얻게 된다. 빌더는 보통 내부 정적 클래스로 만들어 두어서 사용하게 된다.

3. 빌더 패턴


public class NutritionFacts {
	private final int servingSize;
	private final int servings;
	private final int calories;
	private final int fat;
	private final int sodium;
	private final int carbohydrate;

	public static class Builder {
		//필수 매개 변수
		private final int servingSize;
		private final int servings;

		//선택 매개 변수
		private int calories = 0;
		private int fat = 0;
		private int sodium = 0;
		private int carbohydrate = 0;

		public Builder(int servingSize, int servings){
			this.servingSize = servingSize;
			this.servings = servings
		}

		public Builder calories(int val){
			calories = val; return this;
		}
		public Builder fat(int val){
			fat = val; return this;
		}
		public Builder sodium(int val){
			sodium = val; return this;
		}
		public Builder carbohydrate(int val){
			carbohydrate = val; return this;
		}

		public NutritionFacts build(){
			return new NutritionFacts(this);
		}
	}
	private NutritionFacts(Builder builder){
		this.servingSize = builder.servingSize;
		this.servings = builder.servings;
		...
	}
}

// 사용 예시

NutritionFacts pepsi = New NutritionFacts.Builder(40, 8)
  .calorie(100)
  .sodium(35)
  .carbohydrate(27)
  .build();

NutritionFacts 클래스는 불변이며 모든 매개 변수의 기본 값을 한곳에 모아 사용하였다.

장점

빌더에서 setter 메서드는 builder 객체를 반환하게 되는데 이것으로 연쇄적 호출을 사용할 수 있게 되고 좀 더 가독성 있는 코드가 되는 것 같다.

이런 방식을 플루언트 API( fluent API) or 메서드 연쇄(method chaning)이라고 한다.

이 코드의 장점은 쓰기가 쉽고, 무엇보다 읽기가 쉽다는 점에 있다.

빌터 패턴은 (파이썬과 스칼라에 존재하는) 명명된 선택적 매개 변수를 흉내 낸 것이다.

빌더 패턴은 계층적으로 설계된 클래스와 특히 함께 쓰기 좋다.

그렇다면 빌더 패턴의 단점은 무엇이 있을까?

단점

단점으로는 객체를 만들기 위해서 객체보다 빌더를 먼저 만들어 주어야 한다. 물론 빌더를 생성하는 비용이 크지는 않지만 성능이 매우 민감한 상황에서는 문제가 생길 수 있다. 또한 점층적 생성자 패턴의 코드보다는 코드가 장황하기 때문에 매개변수가 4,5개 이상이 되는 경우에만 좀 더 효율이 좋다. 하지만 매개변수가 추가적으로 많아지는 상황이 생기게 되면 기존의 생성자, 정적 팩토리 방식 이였지만 빌더 패턴으로 바꾸는 경우 불편한 경우가 있으므로 추가 변수가 들어 올 수 있다면 빌더 패턴으로 구성하는 것이 좋을 것이다.

결론

  • 생성자나 정적 팩토리로 생성할 때 처리해야 할 파라미터의 갯수가 많은 경우 빌더 패턴을 선택하는 것이 가독성, 확장성 이는 코드가 될 것이다.
  • 매개 변수 중 다수가 필수가 아닌 타입이 아니거나 같은 타입인 경우 특히 빌더 패턴을 고려하자.
  • 빌더는 점층적 생성자보다 가독성이 매우 좋고 쓰기도 좋다. 또한 자바빈즈 패턴 보다 매우 안전하다.

참고

  1. Effective Java 3/e, Joshua Bloch
